- First-run clarity target: planning must describe what the player controls, what is dangerous, what to do next, what success looks like, and why failure happens from the screen itself.
- Feedback/game feel target: every key input, score, near miss, combo, boost, hazard, failure, pause, and retry must define visual/audio/haptic response, timing, and reduced-motion fallback.
- Human visual QA target: QA must judge screenshots/video for readability, motion comprehension, state isolation, and product value, not only object existence or runtime probe fields.
- Blockers: fail if automated checks could pass while a human cannot understand the objective, controls, threat, reward, or retry state from visible evidence.

## 繁體中文

### 範圍
- Enemy Pattern Spec 必須至少定義三個可測的 enemy pattern / hazard pattern：直線壓迫、交錯切換、誘導後封鎖；每個模式都要有 spawn condition、approach motion、warning cue、counterplay、reward/penalty、difficulty ramp 與 QA reproduction step。

### 決策
- 敵人或障礙必須可見地向玩家造成壓力；不能只有靜態方塊、隨機空白、不可預期碰撞或看不出前進方向的物件。

### 交接重點
- 每個 pattern 都要定義最小反應時間、安全通道、危險色、失敗原因文字、near miss 判定與 iPhone SE 2/3 可讀性驗收。

## English

required keywords: enemy, pattern, spec

### Scope
- Enemy Pattern Spec must define at least three testable enemy pattern / hazard pattern entries: direct pressure, alternating weave, and bait-then-block; each needs spawn condition, approach motion, warning cue, counterplay, reward/penalty, difficulty ramp, and QA reproduction step.

### Decisions
- Enemies or hazards must visibly create pressure toward the player; static blocks, random emptiness, unpredictable collisions, or unreadable movement direction fail.

### Handoff Notes
- Each pattern must define minimum reaction time, safe lane, danger color, failure reason text, near-miss rule, and iPhone SE 2/3 readability acceptance.
